**Amends:** Study 002 Pre-Registration, DECISION 002-E (Two-call iteration structure)

---

## Rationale

The original Study 002 pre-registration specified a two-call structure: a routing
analysis pass (Call 1) and a modification decision call (Call 2). The original
design passed all context — routing history, episodic memory, extraction output,
file contents, baseline correction sections, and the response schema — into a
single large modification decision call.

Empirical testing revealed this produces a context overload problem. As iterations
accumulate, the modification decision call grows to contain 9+ prior episodes,
routing history across all iterations, full file contents for multiple files, and
25 abstracts of extraction output simultaneously. Language models do not attend
uniformly across prompts of this size. Early context is compressed and effectively
inaccessible. The routing signal — the component under test in Study 002 — may
fail to influence modification behavior not because it is uninformative but because
the agent cannot attend to it effectively when it is competing with thousands of
tokens of other context.

This is classified as a **baseline correction**, not a change to the component under
test. The routing signal's effect on modification behavior remains what is being
measured. The two-phase call structure removes a confound (context overload) that
would suppress that signal, analogous to how the capability framing correction in
the original pre-registration addressed a different suppression confound from
Study 001. This amendment is committed before the full 20-iteration run begins.

---

## Amended Architecture: Two-Phase Per-Iteration Call Structure

Each iteration now makes two sequential LLM calls. The output of Call 1 feeds into
Call 2 as a compressed, agent-generated summary. The agent reasons about signal data
in Call 1 and reasons about file edits in Call 2. These are cognitively distinct
tasks that benefit from separated context.

---

## Call 1 — Diagnostic Assessment

### Purpose

The agent processes routing history, episodic memory, and extraction output to form
a structured diagnosis of its current situation. No file contents. No edit schema.
Pure signal processing.

### Input

Assembled in this order:

1. **System framing:** The agent is an autonomous research system analyzing its own
   performance trajectory. It must produce a structured JSON assessment of what it
   observes. No edits in this call.

2. **Episodic memory:** All prior episodes in chronological order as a JSON array.
   If empty (iteration 1), communicated explicitly: "This is your first iteration.
   You have no prior episodes."

3. **Routing history:** Full trajectory table (all prior iterations, per-control-abstract
   scores and aggregate) plus the natural language trend statement generated by
   `routing_history.format_for_agent()`. If iteration 1, communicated as: "No prior
   routing scores are available. This is your first modification."

4. **Routing delta:** The consequence of the prior modification — what the last
   edit actually did to routing scores on the 10 control abstracts. Formatted as
   natural language: "After your iteration N modification, routing toward results
   sentences increased on X of 10 control abstracts (aggregate delta: +/-Y)."
   If iteration 1, communicated as: "No prior modification has been made."

5. **Prior extraction output:** The full per-abstract list (abstract_id,
   abstract_text, predicted_claims) from the most recent scanned iteration.
   The agent is told which iteration this is from.

6. **Assessment schema:** The exact JSON structure required, with field descriptions.

### Output Schema

```json
{
  "routing_trend": "improving | declining | flat",
  "last_action_effect": "string",
  "pattern_observed": "string",
  "hypothesis": "string"
}
```

**Field definitions:**

`routing_trend` — Three-way enum reflecting the aggregate routing score direction
across the last 3 completed iterations (or all available iterations if fewer than
3). "improving" means aggregate routing score has net increased. "declining" means
net decreased. "flat" means movement within ±0.02. If iteration 1 (no prior
iterations), value must be "flat".

`last_action_effect` — Plain language description of what the prior modification
did to routing scores. Should reference specific control abstracts where movement
was notable. If iteration 1, value must be: "No prior modification has been made.
This is the first iteration."

`pattern_observed` — What the agent sees across its episode history and routing
trajectory taken together. Should identify whether its modification strategy has
been working, whether routing scores are responding to changes, and whether any
systematic pattern is visible.

`hypothesis` — What the agent thinks should change and why, based on the pattern
observed. This is a forward-looking statement, not an edit instruction. It should
describe a direction ("the prompt needs to be more specific about X" or "the
preprocessing step is not filtering Y correctly") rather than specific file changes.

**Handling iteration 1:** All four fields are required. `routing_trend` is "flat".
`last_action_effect` is the specified string above. `pattern_observed` and
`hypothesis` should reflect reasoning from the extraction output alone, since no
routing history or episode history is available.

**Parse behavior:** Call 1 uses `response_format={"type": "json_object"}`. The
harness parses the response and validates all four fields are present and non-empty.
A malformed or incomplete response is logged as `assessment_malformed` and the
iteration proceeds to Call 2 with a null assessment (Call 2 handles null assessment
gracefully — see below).

## Call 2 — Modification Decision

### Purpose

The agent receives its own compressed diagnosis plus the files it can change, and
produces an episode and edit instructions. No routing history. No extraction output.
No raw episode log. The full history is replaced by the assessment the agent itself
just produced.

### Input

Assembled in this order:

1. **System framing (baseline correction):** The agent is an autonomous extractor
   modifying its own system. Includes full `CAPABILITY_FRAMING`, `WORKED_EXAMPLE_A`,
   `WORKED_EXAMPLE_B`, and `ROUTING_SIGNAL_EXPLANATION` sections from
   `baseline_correction.py`. These are included in every Call 2 at every iteration.

2. **Assessment from Call 1:** The structured JSON object produced by Call 1,
   formatted for readability:

   ```
   YOUR CURRENT ASSESSMENT
   
   Routing trend: [improving | declining | flat]
   
   Effect of last action: [last_action_effect text]
   
   Pattern observed: [pattern_observed text]
   
   Hypothesis: [hypothesis text]
   ```

   If Call 1 returned a null assessment (malformed response), this section reads:
   "Assessment unavailable for this iteration due to a processing error. Proceed
   based on your current file state and the baseline correction guidance above."

3. **Current file contents:** All mutable files keyed by path. Exactly:
   - `prompts/system_prompt.md`
   - `prompts/examples.md`
   - `playground/extractor.py`
   - Any additional `.py` files present in `playground/`

   File contents only. No routing history. No extraction output. No episode log.

4. **Response schema:** The episode + edits JSON structure with field descriptions.

### Output Schema

Unchanged from original pre-registration:

```json
{
  "episode": {
    "observation": "string",
    "hypothesis": "string",
    "action": "string",
    "expectation": "string"
  },
  "rationale": "string",
  "edits": [
    {
      "file_path": "string",
      "operation": "replace_string | replace_file | create_file | delete_file",
      "old_string": "string | null",
      "new_string": "string | null",
      "new_content": "string | null"
    }
  ]
}
```

The episode fields should reflect the agent's reasoning from the assessment. The
`observation` field will naturally reference what the assessment surfaced. The
`hypothesis` field should align with the assessment's `hypothesis` field. These
are the agent's own words — they are not required to match the assessment exactly,
but significant divergence between the assessment hypothesis and the episode
hypothesis is itself an observable worth noting in Sprint 005 analysis.

## Amended `agent_caller.py` Interface

```python
async def invoke_diagnostic(
    prior_output: list[dict],
    prior_output_iteration: int,
    routing_history_text: str,
    routing_delta_text: str,
    prior_episodes: list[dict],
) -> AssessmentResult | AgentFailure

async def invoke_decision(
    assessment: AssessmentResult | None,
    current_files: dict[str, str],
) -> AgentResponse | AgentFailure

async def invoke_repair(
    error_message: str,
    current_files: dict[str, str],
    attempt_number: int,
) -> RepairResponse | AgentFailure
```

`AssessmentResult` dataclass:

```python
@dataclass
class AssessmentResult:
    routing_trend: str          # "improving" | "declining" | "flat"
    last_action_effect: str
    pattern_observed: str
    hypothesis: str
    raw_response: str           # preserved for Sprint 005 analysis
```

`invoke_diagnostic` returns `AssessmentResult` on success or `AgentFailure` on
parse error. `invoke_decision` accepts `AssessmentResult | None` — None triggers
the null assessment handling in the prompt.

## Amended Iteration Loop (study_runner.py)

The iteration loop is amended at step b to split the single agent call into two:

```
b1. Call agent_caller.invoke_diagnostic() with:
    - prior extraction output
    - routing history text
    - routing delta text
    - prior episodes
    → AssessmentResult | AgentFailure

    If AgentFailure:
      log anomaly ("assessment_malformed" or "agent_api_failure")
      assessment = None
      (do NOT skip iteration — proceed to b2 with null assessment)

b2. Call agent_caller.invoke_decision() with:
    - assessment (may be None)
    - current file contents
    → AgentResponse | AgentFailure

    If AgentFailure:
      log anomaly
      write metrics (scanned=false)
      continue to next iteration
```

The diagnostic failure does not halt the iteration. If Call 1 fails, Call 2
proceeds with a null assessment and falls back to file-contents-only reasoning.
This preserves the unconditional N=20 commitment — no iteration is skipped due
to a diagnostic failure alone.

## New Anomaly Types

Added to `anomaly_logger.py`:

- `assessment_malformed` — Call 1 response failed JSON parsing or was missing
  required fields. Assessment set to null. Iteration continues to Call 2.
- `assessment_field_invalid` — A specific field in the assessment had an invalid
  value (e.g., `routing_trend` was not one of the three valid enum values).
  Field is preserved as-is in the record; Call 2 receives the assessment anyway.

## New Metrics Fields

Added to the metrics row:

```json
{
  "assessment_available": true,
  "assessment_routing_trend": "improving | declining | flat | null",
  "call_1_prompt_tokens": 0,
  "call_1_completion_tokens": 0,
  "call_1_tokens_per_second": 0.0,
  "call_2_prompt_tokens": 0,
  "call_2_completion_tokens": 0,
  "call_2_tokens_per_second": 0.0
}
```

`assessment_available` is false when Call 1 returned AgentFailure. The prior
`agent_prompt_tokens`, `agent_completion_tokens`, `agent_tokens_per_second` fields
are now split into call_1 and call_2 variants. The aggregate fields are removed.

## Assessment Persistence

Assessment objects are persisted to a new ledger:
`experiments/study_002/assessments.jsonl`

One line per iteration:

```json
{
  "iteration_n": 3,
  "timestamp": "ISO8601",
  "routing_trend": "declining",
  "last_action_effect": "...",
  "pattern_observed": "...",
  "hypothesis": "...",
  "raw_response": "..."
}
```

This ledger is separate from `episodes.jsonl`. In Sprint 005 analysis, the
assessments and episodes are compared to measure calibration between the agent's
diagnostic reasoning (Call 1) and its stated reasoning in the episode (Call 2).

## Sprint 005 Analysis Addition

This amendment adds one analysis dimension to Sprint 005:

**Assessment-episode alignment:** For each iteration, compare the assessment's
`hypothesis` field (Call 1) with the episode's `hypothesis` field (Call 2). Do
they describe the same modification direction? Divergence between what the agent
diagnosed and what it decided to do is a meaningful observable — it may indicate
that seeing the file contents in Call 2 caused the agent to revise its plan, or
that the assessment reasoning is disconnected from the edit decisions.

**Assessment calibration:** Compare `routing_trend` classifications against the
actual routing score trajectory in `routing_history.jsonl`. Did the agent correctly
identify whether routing was improving, declining, or flat? Assessment accuracy is
a proxy for whether the diagnostic context (Call 1's input) is sufficient for the
agent to form an accurate picture of its own situation.

## Context Size Comparison

| | Original (single call) | Amended (Call 1) | Amended (Call 2) |
|---|---|---|---|
| Routing history | ✓ | ✓ | ✗ |
| Prior episodes | ✓ | ✓ | ✗ |
| Extraction output | ✓ | ✓ | ✗ |
| File contents | ✓ | ✗ | ✓ |
| Baseline correction | ✓ | ✗ | ✓ |
| Assessment | — | ✗ | ✓ |
| Response schema | ✓ | Assessment schema | Edit schema |

Each call carries roughly half the total context. More importantly, each call
carries only the context relevant to its specific cognitive task. The agent
reasoning about routing trends is not simultaneously trying to read Python files.
The agent proposing edits is not simultaneously processing 25 abstracts of
extraction output.
